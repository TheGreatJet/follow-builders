# AI Builders 日报 —— 云端自动采集、提炼与推送搭建指南

每天早上自动把一批 AI 领域 builder 的推特、官方博客、播客,**用大模型整理成一份中文日报**,同时推送到**邮箱**和**微信**。全程跑在 GitHub Actions 上,本地零维护,到点自动发。

本文记录这套系统的完整搭建方法,供复刻与分享。

---

## 0. 它每天在干什么(一句话)

```
原始素材(feed JSON) ──► 大模型(GLM)提炼成中文 HTML 日报 ──► 邮件 + 微信双通道推送
       ▲                                                        ▲
       │                                                        │
   上游 follow 工具定时抓取                              cron-job.org 每天到点触发 GitHub Actions
```

最终产物:一封排版好的中文邮件(到收件箱)+ 一条微信服务号消息(标题 + markdown 正文)。

---

## 1. 信息从哪来:三个 feed 文件

素材集中存放在一个 GitHub 仓库的 `main` 分支,以 JSON 形式提供,通过 `raw.githubusercontent.com` 直接读取:

| 文件 | 内容 | 关键字段 |
|---|---|---|
| `feed-x.json` | AI builder 们的推特 | `.x[]` → `{name, handle, bio, tweets[]}`;tweet → `{text, url, likes}` |
| `feed-blogs.json` | 官方博客文章 | `.blogs[]` → `{name, title, url, author, content}` |
| `feed-podcasts.json` | 播客单集 | `.podcasts[]` → `{name, title, url, transcript}` |

这些 feed 由**上游的 follow 抓取工具**定时更新(维护一份「关注哪些人/博客/播客」的清单,抓全文/转写)。本日报系统只负责**读取这些 JSON 并提炼**,不负责抓取。

> 编辑规则单独放在仓库 `/prompts/` 目录,共 5 个 md 文件:
> `digest-intro.md`(全局格式)、`translate.md`(中译规则)、`summarize-tweets.md`、`summarize-blogs.md`、`summarize-podcast.md`。
> 把「规则」和「代码」分开,改文风不用动代码。

## 2. 如何提炼成日报:大模型 remix

核心脚本 `scripts/digest-email.js`(纯 Node,零依赖,只用内置 `fetch`)。流程:

### 2.1 拉取 + 截断(控制上下文长度)
并行拉取 3 个 feed + 5 个 prompt,带重试(`raw.githubusercontent.com` 偶发 `ECONNRESET`)。为不超 GLM 的 128K 上下文,做预算截断:

| 维度 | 预算 |
|---|---|
| 每个 builder 的推文 | 取点赞最高的 **3** 条 |
| 播客单集 | 最多 **2** 集,每集转写取前 **8000** 字 |
| 博客文章 | 最多 **5** 篇,每篇正文取前 **4000** 字 |

### 2.2 拼 prompt
- **system**:你是《AI Builders 日报》的中文编辑 + 注入 5 个 prompt 规则 + 一组硬性约束(见下)。
- **user**:今日素材(截断后的推文/博客/播客文本)+ feed 生成时间。

### 2.3 硬性约束(写进 system,确保产出稳定)
- **纯中文输出**,技术术语(LLM/GPU/API/RAG/agent/fine-tuning 等)、人名、公司名、产品名保留英文。
- **板块顺序固定**:X / 推特 → 官方博客 → 播客;某板块无内容则整段省略。
- **每条必须有原始 URL**,无 URL 的内容直接丢弃,**绝不编造**。
- **详细度**:每条展开「观点 + 依据」(数据/例子/对比/原话),让读者知道这人讲了什么、凭什么这么讲。**宁可详尽,不要压成一句话摘要。**
- **输出 HTML 片段**(只 body 内部,不要 `<html>/<head>`,不要 ``` 代码块,不要解释文字)。
- **链接规则**:所有原始 URL 一律写成 `<a href="URL">详情</a>`,紧跟摘要末尾同一行,不另起一行,绝不把原始 URL 明文出现。
- **关键词高亮**:每条挑 1–3 个核心概念/数字/产品名,用 `<span class="hl">关键词</span>` 包裹。
- 正文**不用 em-dash(—)**,用中文标点。

### 2.4 结构模板(写死在 prompt 里)
```
<h1>AI Builders 日报 · 日期</h1>
<h2>X / 推特</h2>
  <div class="item"><h3>作者 (角色/公司)</h3><p>核心观点(1句)+ 论据展开(2-4句),高亮关键词。<a href="URL">详情</a></p></div>
<h2>官方博客</h2>
  <div class="item"><h3>博客名: 标题</h3><p>核心主张(1句)+ 关键细节(2-4句)。<a href="URL">详情</a></p></div>
<h2>播客</h2>
  <div class="item"><h3>节目 · 集标题</h3><p>The Takeaway(一句核心收获)+ 展开介绍(2-4句)。<a href="URL">详情</a></p></div>
<p class="footer">由 Follow Builders 技能自动生成 · <a href="仓库URL">详情</a></p>
```

### 2.5 调 GLM
- 端点:`https://open.bigmodel.cn/api/paas/v4/chat/completions`
- 模型:`glm-4.6`(默认,可用仓库变量 `GLM_MODEL` 覆盖)
- 参数:`temperature 0.7`,`top_p 0.9`,`max_tokens 8192`,失败重试 1 次。
- GLM 返回 HTML 片段(偶尔会被 ```html 围栏包住,有个 `sanitizeFragment()` 统一剥掉)。

### 2.6 空内容日(省 token)
如果当天三个板块全空,**跳过 GLM**,直接发一条「今日各板块均无新内容更新」的提示邮件。

## 3. 推送配置:邮件 + 微信双通道

GLM 产出的同一份 HTML,分别转给两个通道:

### 3.1 邮件(Resend)
- 端点:`POST https://api.resend.com/emails`,`Authorization: Bearer <RESEND_API_KEY>`
- `from`: `AI Builders Digest <digest@resend.dev>`(Resend 共享域,只能发到注册邮箱)
- `to`: `DELIVERY_EMAIL`(收件人)
- `subject`: `AI Builders 日报 · <星期 月 日 年>`
- `html`: GLM 产出的 HTML(套一层移动端友好的样式 `wrapHtml`)
- `text`: HTML 降级纯文本(`htmlToText`,把「详情」链接展开成 `文字: URL` 保留可点)

### 3.2 微信(Server酱)
- 端点:`POST https://sctapi.ftqq.com/<SENDKEY>.send`,`Content-Type: application/x-www-form-urlencoded`
- **SendKey 放在 URL 路径里,不需要 header 鉴权**
- 表单字段:`title`(标题)+ `desp`(markdown 正文,限 32KB)
- 成功判定:`code: 0`
- markdown 由 HTML 转换而来(`htmlToMarkdown`):
  - 剥掉 `<head>/<style>`
  - `<a href>` → `[文字](url)`
  - `<h1/2/3>` → `# / ## / ###`
  - `<span class="hl">` → `**加粗**`
  - `<p>/<div>/<li>`/`<br>` → 换行
- **Server酱 把 markdown 渲染成富文本** —— 这就是在微信里看到的「公众号样式」。

### 3.3 双通道容错
邮件是**主通道**(失败直接抛错,整个 run 失败);微信是**附加通道**(`SERVERCHAN_SENDKEY` 没配就跳过,推失败只记日志、不影响已经发出的邮件)。

### 3.4 收件邮箱怎么选
- **公司邮箱会拦外部来源**(Resend 共享域、QQ/163/Gmail SMTP 对公司域都是外部,统统进不去),除非走 IT 把发件域加白名单。
- 实测可行:**用邮箱绑定的手机号邮箱(如 139 邮箱)** —— Resend 共享域正好允许发到注册邮箱,139 稳定收到。

## 4. 调度配置:cron-job.org 触发 GitHub Actions

**不用 GitHub 自带的 `schedule:` cron** —— fork 仓库的 schedule 历史不稳定(经常延迟或漏跑)。改用 **cron-job.org 到点打 `workflow_dispatch`** 触发,稳得多。

### 4.1 GitHub workflow 文件(`.github/workflows/send-digest.yml`)
```yaml
name: Send Daily Digest Email

# 由 cron-job.org 定时打 workflow_dispatch 触发。
# 不用 GitHub schedule —— fork 的 schedule 不稳定。
on:
  workflow_dispatch: {}

permissions:
  contents: read

jobs:
  send:
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - name: Generate and send digest
        working-directory: scripts
        env:
          GLM_API_KEY: ${{ secrets.GLM_API_KEY }}
          RESEND_API_KEY: ${{ secrets.RESEND_API_KEY }}
          DELIVERY_EMAIL: ${{ secrets.DELIVERY_EMAIL }}
          SERVERCHAN_SENDKEY: ${{ secrets.SERVERCHAN_SENDKEY }}
          GLM_MODEL: ${{ vars.GLM_MODEL }}
        run: node digest-email.js
```

### 4.2 cron-job.org 上加一个 job
克隆/新建,填:

| 字段 | 值 |
|---|---|
| **URL** | `https://api.github.com/repos/<你的用户名>/<仓库名>/actions/workflows/send-digest.yml/dispatches` |
| **Method** | `POST` |
| **Header** | `Authorization: Bearer <你的 Actions:Write PAT>` |
| **Header** | `Accept: application/vnd.github+json` |
| **Header** | `Content-Type: application/json` |
| **Body** | `{"ref":"main"}` |
| **时区** | `Asia/Shanghai`(UTC+8)—— 别用默认,否则你设的「9:00」不是北京 9:00 |
| **Schedule** | 每天,挑一个分钟(例如 09:00 左右) |
| **预期响应** | `204 No Content` |

### 4.3 PAT 怎么配
- 用 **fine-grained PAT**:Repository access 里**勾上目标仓库**;Repository permissions 里 **Actions = Read and write**。
- ⚠️ 改 PAT 的仓库/权限范围**不需要重新生成**(token 值不变,cron-job.org 里存的那串继续用)。只有点了 Regenerate 才会换值。
- ⚠️ PAT 的完整值**只在创建时显示一次**,之后 masked,拿不回 —— 所以编辑范围(而非重新生成)是正确做法。

## 5. GitHub 仓库 Secrets 一览

仓库 Settings → Secrets and variables → Actions:

| 名字 | 类型 | 用途 |
|---|---|---|
| `GLM_API_KEY` | secret | BigModel(GLM)API key |
| `RESEND_API_KEY` | secret | Resend API key |
| `DELIVERY_EMAIL` | secret | 收件邮箱 |
| `SERVERCHAN_SENDKEY` | secret | Server酱 SendKey(可选,不配则只发邮件) |
| `GLM_MODEL` | **variable**(非 secret) | 模型名覆盖,默认 `glm-4.6` |

设 secret 命令行(值走自己终端 tty,不经他人):
```bash
gh secret set SERVERCHAN_SENDKEY -R <你的用户名>/<仓库名>
```

## 6. 从零搭建清单(按顺序照做)

1. **准备素材仓库**:fork/建一个仓库,放好 `feed-x.json` / `feed-blogs.json` / `feed-podcasts.json` 和 `/prompts/` 下的 5 个规则文件。
2. **放脚本**:把 `scripts/digest-email.js` 和 `.github/workflows/send-digest.yml` 放进仓库 `main` 分支。
3. **申请各路 key**:
   - BigModel(GLM)API key
   - Resend API key(注册账号、验证收件邮箱)
   - Server酱 SendKey(sct.ftqq.com,微信扫码绑定)
4. **填 GitHub Secrets**:按第 5 节表填 4 个 secret(+ 1 个可选 variable)。
5. **生成 PAT**:fine-grained,勾上仓库 + Actions:Write。
6. **手动验证**:`gh workflow run send-digest.yml -R <owner>/<repo>` 触发一次,确认邮件 + 微信都到。
7. **配 cron-job.org**:按第 4.2 节建 job,**时区设 Asia/Shanghai**,**记得把 job 开关打开(enabled)**。
8. 完事,每天到点自动发。

## 7. 踩过的坑(分享时最有用的部分)

| 坑 | 现象 | 解法 |
|---|---|---|
| **GitHub schedule 不可靠** | fork 仓库的 cron 经常延迟/漏跑 | 改用 cron-job.org 打 `workflow_dispatch` |
| **公司邮箱拦外部邮件** | Resend / SMTP 发到公司域全进不去 | 用手机号邮箱(139);或走 IT 加白名单 |
| **cron-job.org 时区** | 设了「9:00」却迟迟不触发 | job 时区改 `Asia/Shanghai`,默认可能是 UTC |
| **cron-job.org job 没开 enabled** | 到点完全没反应 | 新建 job 后确认开关是开的 |
| **fine-grained PAT 漏勾仓库** | dispatch 返回 404,Actions 没有任何 run 痕迹 | PAT 的 Repository access 勾上目标仓库 + Actions:Write |
| **GLM 把 HTML 包在 ``` 里** | 邮件正文出现代码围栏 | `sanitizeFragment()` 统一剥掉 |
| **raw.githubusercontent 偶发 ECONNRESET** | 拉素材随机失败 | `fetchWithRetry()` 重试 3 次 |
| **Server酱 desp 32KB 上限** | 超长正文被截断 | 控制日报长度;超长需分段 |
| **Server酱 成功判定** | 不是看 HTTP 200,是看 `code:0` | 判断 `data.code !== 0` 即失败 |
| **Node 20 即将下线** | checkout/setup-node@v4 有 deprecation 告警 | 后续升 @v5(非阻塞) |

## 8. 设计要点小结

- **采集 / 规则 / 代码 三分离**:feed 是数据,prompts 是文风,脚本是流程。改一样不动另两样。
- **大模型只做提炼,不做采集**:保证「绝不编造」,所有内容必须有原始 URL。
- **双通道容错**:邮件主、微信辅,微信挂了不耽误邮件。
- **外部调度 > 内置调度**:cron-job.org 比 fork 的 schedule 可靠得多。
- **零本地维护**:全在 GitHub Actions 跑,本地只读 PAT,改文件走网页/Contents API。
